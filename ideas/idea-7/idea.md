# Idea 7: Memory Budget Annotations — Inline Allocation Caps Verified by CI Across All Languages

## One-line summary
Embed structured memory-budget annotations in hot-path code that AI tools must honor and Jenkins benchmarks verify per PR.

## Tags
density-research, multi-language, jenkins, shared-protocol, perf-regression

## Problem
Density research lives or dies by memory discipline in hot paths. A C++ allocator function has an implicit budget — "this should use ≤2MB RSS per shard" — but that budget exists only in a developer's head or a Slack thread. AI tools freely generate code in these hot paths without knowing the budget exists. A Copilot completion adds a vector pre-allocation that blows the budget by 3×; a Claude Code refactor introduces a map where a fixed array would fit within budget. The regression is caught days later in a nightly benchmark, long after the PR merged.

The team has per-service resource requests in Helm charts, but those are coarse (whole-pod level). There is no per-function or per-module budget that AI tools can read, respect, and that CI can verify at the code level.

## Proposal
Define a structured annotation format that developers embed directly in source code as structured comments. Each annotation declares a memory budget for a function, module, or allocation site. The format is language-agnostic but syntax-adapted:

- Go: `// @mem-budget: max_rss=2MB scope=function bench=go_test`
- C++: `// @mem-budget: max_rss=512KB scope=block bench=allocator_bench`
- Python: `# @mem-budget: max_heap=50MB scope=module bench=pytest_memray`
- Node.js: `// @mem-budget: max_heap=100MB scope=function bench=clinic_heap`
- C: `/* @mem-budget: max_rss=1MB scope=function bench=valgrind_massif */`

A registry file `.mem-budgets/index.jsonl` is auto-generated by `scripts/extract-mem-budgets.sh`, which scans the codebase for `@mem-budget` annotations and produces a structured index. All four AI tool rule files include the directive: "Before modifying any function with a `@mem-budget` annotation, read the budget and ensure your changes respect it. If unsure, add a comment flagging the risk."

A Jenkins stage `mem-budget-verify` runs benchmarks for any function whose `@mem-budget` annotation is in the PR diff's changed files. It compares observed RSS/heap against the declared budget and fails the PR if the budget is exceeded without an explicit waiver (`@mem-budget-waiver: reason="...""`).

## Cross-AI-tool design

### Claude Code
- **Role:** Budget author and density optimizer. Claude Code's terminal-native agentic mode is ideal for profiling-informed budget setting: run a benchmark, observe RSS, set the `@mem-budget` annotation. Also the primary tool for density-research refactors that must stay within budgets. A custom command (`/.claude/commands/check-mem-budgets.md`) runs the extraction and verification locally.
- **Shared contract it reads/writes:** Reads/writes `@mem-budget` annotations in source files. Reads `.mem-budgets/index.jsonl`. Reads `CLAUDE.md` budget-awareness rules.
- **Fallback if unavailable:** Developer sets annotations manually using the documented format. Jenkins verifies regardless.

### GitHub Copilot
- **Role:** Budget-aware inline completions. `.github/copilot-instructions.md` includes the rule: "Respect `@mem-budget` annotations. Do not generate allocations that could exceed the declared limit." Copilot Coding Agent PRs that touch annotated functions must not introduce new allocations without budget justification. `.github/prompts/mem-budget-check.prompt.md` provides a verification prompt.
- **Shared contract it reads/writes:** Reads `@mem-budget` annotations in the current file context. Reads `.mem-budgets/index.jsonl` for cross-file awareness.
- **Fallback if unavailable:** Jenkins `mem-budget-verify` catches violations. The annotations are visible to any tool or human.

### Codex
- **Role:** Pre-submit budget verifier. `AGENTS.md` instructs Codex to run `scripts/verify-mem-budgets.sh --changed` before submitting any task that modifies files with `@mem-budget` annotations. Codex executes the listed benchmark command and checks results against the declared budget.
- **Shared contract it reads/writes:** Reads `AGENTS.md` directive, `@mem-budget` annotations, and `.mem-budgets/index.jsonl`. Writes benchmark results to a local report.
- **Fallback if unavailable:** Jenkins is the definitive verifier. Local pre-submit is a speed optimization.

### Cursor
- **Role:** IDE budget visualizer. `.cursor/rules/mem-budgets.mdc` instructs Cursor to surface `@mem-budget` annotations prominently when editing annotated functions. A custom command `/mem-budget` shows the budget, the benchmark command, and the last known measurement for the current function. Cursor's inline editing makes it the best surface for "you're about to exceed budget" warnings.
- **Shared contract it reads/writes:** Reads `@mem-budget` annotations and `.mem-budgets/index.jsonl`. Displays budget info in IDE.
- **Fallback if unavailable:** `scripts/show-mem-budget.sh <file:line>` from the terminal.

## Languages affected
All five: Go (service-level RSS budgets for density), C++ (allocator and arena budgets — the core density surface), C (driver/primitive memory budgets), Python (data pipeline heap budgets via memray), Node.js (BFF heap budgets via clinic). Each language uses its native comment syntax but the same `@mem-budget` key-value format. Benchmark commands are language-specific (go test -bench, allocator_bench, pytest+memray, clinic heapprofile, valgrind massif).

## Infra impact
- **Jenkins:** New stage `mem-budget-verify` in the shared library. Runs after `build`, before `test`. Executes benchmarks only for changed annotated functions (scoped, not full suite). Adds 1-5 minutes per PR depending on benchmark count. Blocking on budget exceedance without waiver.
- **Kubernetes:** No direct runtime impact, but budgets correlate to Helm resource requests — a future integration could auto-suggest request adjustments when budgets change.
- **Density/perf research:** Core density mechanism. Budgets encode the density team's targets at the code level, making them enforceable by CI and visible to AI tools. This is the closest coupling between AI tooling and density research in this repo.

## Concrete artifact

Source annotation (Go):
```go
// @mem-budget: max_rss=2MB scope=function bench="go test ./services/shard/... -bench=BenchmarkAllocBurst -run=^$ -benchmem"
func AllocBurstPool(arena *jemalloc.Arena, n int) (*Pool, error) {
    // ...
}
```

`.mem-budgets/index.jsonl` (auto-generated):
```jsonl
{"file":"services/shard/alloc.go","line":42,"function":"AllocBurstPool","lang":"go","max_rss":"2MB","bench":"go test ./services/shard/... -bench=BenchmarkAllocBurst -run=^$ -benchmem","last_measured":"1.7MB","last_ts":"2026-04-15"}
{"file":"lib/alloc/burst_pool.cc","line":18,"function":"burst_pool_alloc","lang":"cpp","max_rss":"512KB","bench":"./bench/allocator_bench --scenario=burst","last_measured":"480KB","last_ts":"2026-04-14"}
```

Jenkins stage:
```groovy
stage('mem-budget-verify') {
  steps {
    script {
      def results = memBudgetVerify(index: '.mem-budgets/index.jsonl', diffAgainst: 'origin/main')
      results.each { r ->
        if (r.exceeded && !r.waiver) { error("Memory budget exceeded: ${r.function} used ${r.observed} > ${r.budget}") }
      }
      archiveArtifacts artifacts: '.mem-budgets/results/*.json', onlyIfSuccessful: false
    }
  }
}
```

## Success metric
- 50+ `@mem-budget` annotations in density-critical paths within 8 weeks, covering ≥80% of functions flagged in density-research benchmarks.
- Budget exceedances caught pre-merge increase from 0 (current) to ≥5/month, each representing a prevented density regression.
- Post-merge density regressions in annotated paths drop to zero within one quarter.
- AI tools generate budget-compliant code in annotated functions ≥90% of the time, measured by Jenkins pass rate on `mem-budget-verify`.

## Risks & failure modes
- **Annotation rot:** Budgets become stale as workloads change. Mitigated by storing `last_measured` in the index and flagging annotations whose last measurement is >30 days old.
- **Benchmark flakiness:** Memory measurements vary between runs. Mitigated by requiring multiple samples and using p95 as the comparison point, plus a configurable tolerance margin.
- **Over-annotation:** Developers annotate everything, diluting value. Mitigated by guidelines: annotate only density-critical hot paths (top-50 functions by RSS contribution).
- **Tool compliance fatigue:** AI tools may ignore budget annotations under context pressure. Mitigated by the Jenkins gate — compliance is enforced at CI, not just requested.

## Originality statement

**Versus existing ideas in this repo:**
Idea 2 (Canary Identifiers) uses annotations to trap hallucinations; idea 3 (Perf Witness Tickets) uses per-PR prediction tickets. This idea uses inline code annotations for per-function memory budgets — a different granularity (function vs. PR), different metric (RSS/heap vs. latency/density), and different mechanism (inline annotation vs. sidecar file). Overlap with idea 3: ~20% (both enforce perf discipline via CI). Overlap with idea 2: ~10%.

**Versus common industry proposals:**
This is not a lint agent, test generator, or generic performance dashboard. Performance budgets exist in web frontend development (Lighthouse performance budgets) but applying per-function memory budgets as structured source-code annotations that AI tools read, respect, and CI verifies is novel. The closest pattern is compiler pragmas (`#pragma` for alignment, etc.), but those are enforced at compile time, not CI benchmark time, and don't involve cross-tool coordination. It satisfies novelty heuristics #1 (new inter-tool annotation protocol), #3 (density-research coupling — this is the density tool), #4 (Jenkins leverage), and #5 (language-stack specificity — five languages, five benchmark tools).
