# Idea 19: Diff Provenance Chain — Per-Hunk Tool Attribution in AI-Authored Diffs

## One-line summary
Annotate each diff hunk with the specific AI tool and prompt that generated it, enabling per-hunk review routing and granular tool-quality metrics.

## Tags
shared-protocol, shared-evaluation, cross-tool-routing, jenkins

## Problem
The `AI-Tool:` commit trailer attributes an entire commit to one tool, but real-world AI-assisted development is finer-grained. A single PR may contain hunks from three different tools: Copilot wrote the handler, Claude Code refactored the allocator, and Cursor fixed the tests. The commit trailer says "copilot" (the last tool used), hiding the contributions of the others.

This granularity loss has three consequences: (1) tool-quality metrics (from ideas 2, 11, 14) are noisy because they attribute whole commits when only specific hunks may be problematic, (2) review effort is misallocated because a reviewer can't tell which hunks came from which tool (and should be scrutinized differently), and (3) the team can't answer "which tool wrote the code that caused this bug?" at the hunk level.

## Proposal
Introduce a diff provenance annotation system that tracks tool attribution at the hunk level:

1. **Annotation format:** Each AI-generated hunk is preceded by a structured comment: `// @ai-provenance: tool=claude-code prompt="refactor AllocBurstPool to use arenas" ts=2026-04-19T10:00:00Z`. These annotations are machine-parseable and stripped before final merge (recorded in the provenance ledger).
2. **Provenance ledger:** `.provenance/ledger.jsonl` stores the provenance chain for every AI-authored hunk: file, line range, tool, prompt summary, timestamp, and outcome (CI pass/fail, review comments, post-merge survival).
3. **Pre-merge stripping:** A pre-merge hook (`scripts/strip-provenance-comments.sh`) removes `@ai-provenance` annotations from the code, records them in the ledger, and commits the cleaned code. The annotations exist during development and review but don't pollute the final codebase.
4. **Review routing:** During review, provenance annotations guide the reviewer: "This hunk was generated by Copilot inline completion — check for hallucinated function calls" vs. "This hunk was refactored by Claude Code — check for semantic preservation."

## Cross-AI-tool design

### Claude Code
- **Role:** Automatic provenance emitter. Claude Code's session tracking makes it natural to tag every output hunk with provenance. `CLAUDE.md` includes: "Prepend `@ai-provenance` to every non-trivial code block you generate." Claude Code also generates the weekly provenance analysis (per-tool hunk quality breakdown).
- **Shared contract it reads/writes:** Writes `@ai-provenance` annotations in code. Reads `.provenance/ledger.jsonl` for analysis.
- **Fallback if unavailable:** Manual annotation using the format. The ledger and stripping work regardless.

### GitHub Copilot
- **Role:** Inline provenance via prompt file. For inline completions, provenance is harder (Copilot can't add a comment before each completion). `.github/copilot-instructions.md` includes the annotation directive for Coding Agent PRs. For inline completions, a `.github/prompts/annotate-provenance.prompt.md` lets the developer bulk-annotate after accepting a batch of completions.
- **Shared contract it reads/writes:** Writes `@ai-provenance` in Coding Agent PRs. Reads ledger for self-reporting.
- **Fallback if unavailable:** Developer annotates manually or hunks are attributed to "unknown."

### Codex
- **Role:** Clean provenance emitter. `AGENTS.md` directs Codex to add `@ai-provenance` to every generated hunk. Codex's task-file determinism means its provenance annotations are consistent and complete — every hunk is tagged.
- **Shared contract it reads/writes:** Writes `@ai-provenance` annotations. Reads `AGENTS.md`.
- **Fallback if unavailable:** Hunks from Codex are attributed at commit level via trailer. Hunk-level detail is lost but the system still works.

### Cursor
- **Role:** IDE provenance overlay. `.cursor/rules/provenance.mdc` instructs Cursor to add `@ai-provenance` annotations during Agent/Composer mode. A custom command `/provenance` displays provenance annotations as an IDE overlay during review, showing tool attribution per hunk without the annotations cluttering the view. Cursor's inline display is ideal for "highlight which tool wrote which hunk" during review.
- **Shared contract it reads/writes:** Writes `@ai-provenance` in Agent mode. Reads annotations for IDE display.
- **Fallback if unavailable:** Annotations are plain comments visible in any editor.

## Languages affected
Language-agnostic annotation format (structured comment using each language's syntax). The provenance protocol works identically for Go, Python, Node.js, C++, and C code. The ledger and stripping are language-aware only for comment syntax. Most valuable for cross-language PRs where hunks span multiple tools and languages.

## Infra impact
- **Jenkins:** Pre-merge hook strips annotations and writes to ledger. New optional stage `provenance-record` ensures all annotations are captured before stripping. Lightweight (<10s). No blocking behavior — provenance is informational.
- **Kubernetes:** No runtime impact.
- **Density/perf research:** Indirect — enables hunk-level attribution for density-critical code, so when a density regression is found, the team can attribute it to the specific hunk and tool, not just the commit.

## Concrete artifact

Source annotation (during development/review):
```go
// @ai-provenance: tool=claude-code prompt="refactor AllocBurstPool to use jemalloc arenas" ts=2026-04-19T10:00:00Z
pool := arena.NewPool(arena.WithBurstSize(2 * 1024 * 1024))

// @ai-provenance: tool=copilot prompt="inline completion" ts=2026-04-19T10:05:00Z
if err := pool.Warmup(ctx); err != nil {
    return nil, fmt.Errorf("warmup failed: %w", err)
}
```

`.provenance/ledger.jsonl` (after stripping):

## Success metric
- ≥50% of AI-authored hunks carry provenance annotations within 8 weeks.
- Hunk-level attribution enables at least 3 per-tool insights per quarter that commit-level attribution would have missed.
- Review routing based on provenance reduces review time on low-risk hunks by 20%.
- The stripping process introduces zero code defects (annotations are cleanly removed in 100% of cases).

## Risks & failure modes
- **Annotation compliance:** Tools may not consistently add provenance. Mitigated by treating unannotated hunks as "unknown" and tracking annotation rate as a metric.
- **Annotation noise:** Too many annotations clutter the code during review. Mitigated by the IDE overlay approach (Cursor) and by keeping annotations as single-line comments.
- **Stripping errors:** The strip script may accidentally remove or corrupt code. Mitigated by running the strip in a sandboxed diff check (strip + compile + test) before committing.
- **Privacy concerns:** Prompt text in annotations may reveal developer intent. Mitigated by allowing "prompt=redacted" for sensitive tasks.
